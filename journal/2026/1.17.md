## 💡 今日份“顿悟”

今天最大的冲击是彻底扭转了对 Agent 的理解。

原来面试官看重的不是我会调用多少模型 API，而是我能不能把不稳定的 LLM 封装成一个“可控、可验证、可落地”的产品。

我之前一直想着怎么让模型更聪明，其实工程的本质是：把模型当做不可信组件，用状态机、工具白名单和 Schema 把它“管”起来。这正好是我们前端最擅长的交互和流程控制。

## 🧠 知识内化（我的理解）

### 1) Agent 的本质

以前我觉得 Agent 就是个“高级聊天机器人”，会自动思考。现在明白了，工程视角下的 Agent 其实是“状态机 + 工具调用”。

- **LLM 不是大脑，它是 CPU**：它没有记忆（要靠我们存），没有手脚（要靠 Tool）。
- **受控才是灵魂**：不能让它随便聊，要给它“白名单工具”和“结构化输出（JSON Schema）”。
- **人机协作**：不要指望 AI 一次全对，关键是在流程里设计“确认/编辑”的节点，让用户兜底。

### 2) 技术选型与部署

为了秋招作品集，我必须做一个“端到端闭环”，而不是简单的 Demo。

- **前端栈**：果断选 Next.js（React）。虽然我 Vue 熟，但为了适应大厂和完整的产品闭环，Next.js 是最优解。
- **数据流**：Supabase（Auth + Postgres + Storage）解决后端和跨设备同步，太适合我这种独立开发者了。
- **部署**：不上 Vercel，直接用 Docker + 1Panel + 腾讯云。这能体现我的运维和 Docker 能力，面试时能吹一波“私有化部署”和“反向代理”。
- **地图**：高德（AMap）。因为是国内落地项目，必须处理好 GCJ-02 坐标转换，这些细节才是工程亮点。

### 3) 主项目：PhotoCoach Agent

不做泛泛的助手，专攻“摄影 + 旅行记录”这个垂直场景。

- **核心逻辑**：照片导入 → EXIF 解析 → 地点确认/推测 → 风格化总结 → 导出。
- **亮点**：这不是聊天，是一个工作流。无 GPS 时的“推测 + 人工确认”、生成后的“Markdown 编辑”，这些都是前端工程价值的体现。
